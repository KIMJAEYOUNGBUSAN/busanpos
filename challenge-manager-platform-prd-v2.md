# 챌린지 매니저 플랫폼 PRD v2

작성일: 2026-06-11  
문서 목적: 실제 구현 가능한 1차 릴리즈 범위를 확정한다.  
참고 문서: `challenge-manager-platform-plan.md`, `challenge-manager-platform-prd.md`  
문서 결론: MVP는 "자동 결제/자동 상금 지급 플랫폼"이 아니라 "챌린지 인증, 검수, 랭킹 확정, 정산 보조 플랫폼"으로 구현한다.

## 1. 재분석 결론

기존 기획서와 1차 PRD는 기능 범위가 충분하지만, 실제 개발에 바로 들어가기에는 아직 넓다. 특히 결제, 상금, 외부 API 자동 검수, 세금, 광고 표시, 개인정보 처리는 초기 제품의 리스크를 크게 만든다.

따라서 1차 릴리즈는 다음 원칙으로 제한한다.

- PG 결제 연동 없음
- 상금 자동 송금 없음
- 참가비 자동 재분배 없음
- 외부 플랫폼 API 자동 검수 없음
- 세금 자동 계산 없음
- 커뮤니티 기능 없음
- 앱 푸시, 문자, 카카오 알림톡 없음
- 운영자가 수동으로 결제 상태와 지급 상태를 관리
- 참가자는 링크와 사진으로 인증
- 운영자는 제출물을 수동 검수
- 시스템은 승인된 제출물 기준으로 진행률, 성공 여부, 랭킹을 계산
- 운영자는 최종 정산 대상자 CSV를 다운로드

## 2. 제품 한 줄 정의

운영자가 블로그, 유튜브, 틱톡 챌린지를 만들고 참가자의 링크/사진 인증을 검수해 성공자, 랭킹, 정산 대상자를 확정하는 웹 기반 챌린지 운영 도구.

## 3. 1차 릴리즈 목표

## 3.1 비즈니스 목표

- 운영자가 스프레드시트 없이 챌린지 참가자와 인증 제출물을 관리하게 한다.
- 참가자가 본인의 제출 상태, 승인 상태, 진행률을 명확히 확인하게 한다.
- 운영자가 수동 검수만으로도 챌린지 종료 후 성공자와 랭킹을 확정하게 한다.
- 법무, 세무, PG 리스크가 큰 자동 지급 기능 없이 초기 시장 반응을 검증한다.

## 3.2 사용자 목표

운영자:

- 챌린지를 빠르게 개설한다.
- 참가자 결제 상태를 수동으로 관리한다.
- 검수 대기 제출물을 빠르게 처리한다.
- 성공자와 지급 대상자를 실수 없이 확인한다.

참가자:

- 참가 조건을 명확히 확인한다.
- 링크와 사진으로 쉽게 인증한다.
- 승인, 반려, 보완 요청 상태를 확인한다.
- 현재 진행률과 순위를 확인한다.

## 4. 공식 자료 기반 범위 판단

이번 PRD는 다음 공식 자료를 기준으로 리스크를 다시 판단했다.

- YouTube Data API는 API 호출별 quota 비용이 있고, 프로젝트 기본 quota와 감사/확장 절차가 존재한다. 따라서 MVP에서는 유튜브 자동 통계 수집을 제외한다.
- TikTok API는 영상 목록 조회 API와 rate limit, 권한 scope, 접근 제한이 있다. 따라서 MVP에서는 틱톡 자동 검수를 제외한다.
- 개인정보 처리방침 작성지침은 개인정보 처리 목적, 항목, 보유 기간 등을 투명하게 공개하는 것을 요구한다. 따라서 MVP에서는 고위험 개인정보 수집을 피하고 동의 기록을 남긴다.
- 공정거래위원회 경제적 이해관계 표시 안내는 광고성 추천/보증 콘텐츠의 표시 필요성을 강조한다. 따라서 브랜드 챌린지 또는 리워드성 콘텐츠는 광고 표시 확인 항목을 넣는다.

## 8. 핵심 플로우

## 8.1 챌린지 생성 플로우

1. 운영자가 챌린지 생성 화면에 진입한다.
2. 제목, 설명, 플랫폼 유형, 기간, 모집 인원, 미션 수, 성공 기준을 입력한다.
3. 제출 방식과 검수 기준을 입력한다.
4. 광고 표시 필요 여부를 선택한다.
5. 보상 방식을 선택한다.
6. 환불 정책 문구를 입력한다.
7. 저장한다.
8. 공개 상태로 전환한다.

## 8.2 참가 신청 플로우

1. 참가자가 챌린지 상세를 본다.
2. 참가 조건, 성공 기준, 보상 방식, 환불 정책을 확인한다.
3. 필수 동의 체크박스를 선택한다.
4. 참가 신청을 완료한다.
5. 상태는 `PENDING_PAYMENT`가 된다.
6. 운영자가 수동으로 `PAID` 처리한다.
7. 참가 상태는 `JOINED`가 된다.

## 8.3 제출 및 검수 플로우

1. 참가자가 내 챌린지에서 미션 제출을 선택한다.
2. 링크, 이미지, 설명을 입력한다.
3. 제출 전 체크리스트를 확인한다.
4. 제출 상태는 `SUBMITTED`가 된다.
5. 운영자가 제출물을 확인한다.
6. 운영자는 `APPROVED`, `REJECTED`, `NEEDS_REVISION` 중 하나를 선택한다.
7. 승인 시 진행률에 반영된다.
8. 반려 또는 보완 요청 시 참가자에게 사유가 표시된다.

## 8.4 종료 및 정산 보조 플로우

1. 챌린지 종료일이 지나면 상태가 `REVIEWING`이 된다.
2. 운영자가 남은 제출물을 모두 검수한다.
3. 시스템이 성공 여부와 랭킹을 계산한다.
4. 운영자가 랭킹을 확정한다.
5. 지급 대상자 목록이 생성된다.
6. 운영자가 지급 예정 금액을 입력한다.
7. 운영자가 CSV를 다운로드한다.
8. 외부에서 지급 처리 후 지급 상태를 수동으로 변경한다.

## 9. 기능 상세 요구사항

## 9.1 회원가입과 로그인

요구사항:

- 이메일, 비밀번호, 이름, 닉네임으로 가입한다.
- 이메일은 중복될 수 없다.
- 비밀번호는 해시로 저장한다.
- 로그인 성공 시 세션 또는 토큰을 발급한다.

수용 기준:

- 중복 이메일 가입이 차단된다.
- 로그인하지 않은 사용자는 보호된 화면에 접근할 수 없다.
- 역할에 따라 접근 가능한 메뉴가 다르다.

## 9.2 챌린지 생성

필수 입력:

- 제목
- 설명
- 플랫폼 유형
- 모집 시작일
- 모집 종료일
- 진행 시작일
- 진행 종료일
- 최대 참가자 수
- 총 미션 수
- 최소 승인 미션 수
- 제출 가이드
- 검수 기준
- 보상 방식
- 환불 정책
- 광고 표시 필요 여부

수용 기준:

- 모집 종료일은 모집 시작일보다 늦어야 한다.
- 진행 종료일은 진행 시작일보다 늦어야 한다.
- 최소 승인 미션 수는 총 미션 수보다 클 수 없다.
- 최대 참가자 수는 1명 이상이어야 한다.
- 필수 입력이 없으면 저장할 수 없다.

## 9.3 챌린지 목록

요구사항:

- 공개 상태의 챌린지만 참가자에게 노출한다.
- 플랫폼 유형 필터를 제공한다.
- 모집 중, 진행 중, 종료 상태를 표시한다.

수용 기준:

- 비공개 챌린지는 참가자 목록에 보이지 않는다.
- 모집 종료된 챌린지는 참가 신청할 수 없다.
- 최대 인원 도달 시 참가 신청할 수 없다.

## 9.4 참가 신청과 동의

요구사항:

- 참가 신청 전에 필수 동의를 받는다.
- 동의 버전과 시각을 저장한다.
- 참가 신청 직후 결제 상태는 `PENDING_PAYMENT`다.

필수 동의:

- 서비스 이용 약관
- 개인정보 수집 및 이용
- 챌린지 조건 확인
- 환불 정책 확인
- 광고 표시 안내 확인

수용 기준:

- 필수 동의가 없으면 신청할 수 없다.
- 동일 챌린지에 중복 신청할 수 없다.
- 동의 기록은 삭제하지 않고 보관한다.

## 9.5 결제 상태 수동 관리

요구사항:

- 운영자는 참가자의 결제 상태를 수동 변경한다.
- 실제 금전 이동은 플랫폼 밖에서 처리한다.
- 결제 상태 변경은 로그로 남긴다.

상태:

- `PENDING_PAYMENT`
- `PAID`
- `CANCELED`
- `REFUNDED`

수용 기준:

- `PAID` 상태 참가자만 제출할 수 있다.
- 결제 상태 변경자는 로그에 기록된다.
- 참가자는 본인의 결제 상태를 볼 수 있다.

## 9.6 미션 제출

요구사항:

- 참가자는 미션 번호를 선택해 제출한다.
- 제출물에는 링크 또는 이미지 중 하나 이상이 필요하다.
- 이미지는 최대 5개까지 첨부한다.
- 이미지 파일은 JPG, PNG, WEBP만 허용한다.
- 파일당 최대 용량은 10MB다.

수용 기준:

- 결제 완료 참가자만 제출할 수 있다.
- 같은 미션 번호에 여러 번 제출할 수 있지만, 최신 제출물을 기준으로 검수한다.
- 이전 제출물은 이력으로 보관한다.
- 참가자는 본인 제출물만 볼 수 있다.

## 9.7 제출물 검수

요구사항:

- 운영자는 제출물 목록을 검수 상태별로 필터링한다.
- 운영자는 링크를 새 창으로 열어 확인한다.
- 운영자는 첨부 이미지를 확인한다.
- 승인, 반려, 보완 요청을 선택한다.
- 반려와 보완 요청은 사유가 필수다.

수용 기준:

- 승인된 제출물만 성공 계산에 반영된다.
- 상태 변경마다 검수 로그가 생성된다.
- 참가자는 검수 결과와 사유를 확인할 수 있다.

## 9.8 진행률과 성공 여부

계산식:

- 승인 수 = `APPROVED` 제출물 중 미션 번호별 최신 승인 수
- 진행률 = 승인 수 / 총 미션 수
- 성공 = 승인 수 >= 최소 승인 미션 수

수용 기준:

- 승인 상태가 바뀌면 진행률이 갱신된다.
- 운영자는 참가자별 승인 수를 볼 수 있다.
- 참가자는 본인 진행률을 볼 수 있다.

## 9.9 랭킹

MVP 공식:

1. 승인 수 내림차순
2. 마지막 승인 시각 오름차순
3. 반려 수 오름차순
4. 참가 신청 시각 오름차순

수용 기준:

- 랭킹은 검수 상태 변경 후 다시 계산된다.
- 운영자는 랭킹 확정 버튼을 누를 수 있다.
- 랭킹 확정 후 검수 변경은 관리자 확인이 필요한 예외 작업으로 처리한다.

## 9.10 정산 보조

요구사항:

- 랭킹 확정 후 지급 대상자 목록을 생성한다.
- 지급 예정 금액은 운영자가 수동 입력한다.
- 지급 상태는 운영자가 수동 변경한다.
- CSV 다운로드를 제공한다.

CSV 필드:

- challengeId
- challengeTitle
- participantId
- participantName
- participantEmail
- finalRank
- finalResult
- paymentStatus
- expectedAmount
- payoutStatus
- payoutMemo

수용 기준:

- 실패자는 기본적으로 지급 대상자에서 제외된다.
- 운영자는 지급 보류 상태를 설정할 수 있다.
- 지급 상태 변경은 로그로 남는다.

## 9.11 광고 표시 체크

요구사항:

- 챌린지 생성 시 광고 표시 필요 여부를 설정한다.
- 광고 표시가 필요한 챌린지는 제출 화면에 안내 문구를 노출한다.
- 제출 전 체크리스트에 광고 표시 확인 항목을 포함한다.
- 운영자 검수 화면에도 광고 표시 확인 체크 항목을 노출한다.

수용 기준:

- 광고 표시 필요 챌린지에서 참가자는 체크리스트 확인 후 제출한다.
- 운영자는 검수 시 광고 표시 여부를 확인할 수 있다.

## 9.12 이의 신청

MVP 정식 릴리즈에 포함한다. Alpha와 Beta에서는 제외 가능하다.

- 운영자는 이의 신청에 답변한다.
- 상태는 접수, 검토 중, 처리 완료로 관리한다.

수용 기준:

- 참가자는 본인 관련 이의만 등록할 수 있다.
- 운영자는 본인 챌린지 이의만 답변할 수 있다.

## 18. 수익화 가능성 분석

## 18.1 결론

이 플랫폼은 수익화 가능성이 있다. 다만 수익화의 중심을 "참가비에서 수수료를 떼는 구조"로 시작하면 법무, 세무, PG, 신뢰 리스크가 커진다. 초기 수익화는 참가자가 아니라 챌린지를 운영하는 운영자, 브랜드, 마케팅 대행사, 교육 사업자에게 과금하는 B2B SaaS 구조가 더 현실적이다.

수익화 가능성이 있는 이유는 다음과 같다.

- 브랜드와 광고주는 블로그, 유튜브, 틱톡 등 크리에이터 콘텐츠를 계속 활용하고 있다.
- 크리에이터 캠페인은 모집, 인증, 검수, 리포트, 지급 관리가 번거롭다.
- 소규모 브랜드와 교육 사업자는 대형 인플루언서 플랫폼보다 가볍고 저렴한 운영 도구가 필요하다.
- 챌린지는 단순 광고보다 참가자 행동을 유도하므로 교육, 습관 형성, 콘텐츠 마케팅, 체험단 운영에 적합하다.
- 운영자는 챌린지를 반복 운영할수록 템플릿, 검수, 랭킹, 정산 기능에 비용을 지불할 가능성이 있다.

공식 및 시장 자료 기준으로도 크리에이터 광고와 인플루언서 마케팅은 성장 중이다. IAB는 2025년 미국 크리에이터 광고 지출이 370억 달러에 이를 것으로 전망했고, Grand View Research는 글로벌 크리에이터 이코노미 시장이 2026년 3,104억 달러 규모에서 2033년 1조 3,455억 달러까지 성장할 것으로 전망한다. 이 수치는 플랫폼이 바로 큰 시장을 점유한다는 뜻은 아니지만, 브랜드와 운영자가 크리에이터 캠페인 관리 도구에 돈을 쓸 명분이 있다는 근거가 된다.

- 블로그 체험단 관리
- 유튜브 쇼츠 캠페인 운영
- 틱톡 해시태그 캠페인 운영
- 광고 표시 누락 방지
- 제출 링크와 캡처 자료 정리
- 성과 리포트 확보

과금 가능성:

- 높음

이유:

- 캠페인 운영은 매출과 직접 연결될 수 있다.
- 운영 시간이 줄어들면 비용 절감 효과가 명확하다.
- 광고 표시, 검수 로그, 리포트는 브랜드 입장에서 리스크 관리 가치가 있다.

### 2순위: 마케팅 대행사

니즈:

- 여러 브랜드 챌린지 동시 운영
- 참가자 모집과 제출물 관리
- 클라이언트 제출용 리포트
- 운영자별 권한 관리
- CSV 다운로드와 증빙 보관

과금 가능성:

- 높음

이유:

- 대행사는 반복적으로 캠페인을 운영한다.
- 수동 관리 시간이 곧 인건비다.
- 여러 캠페인을 관리하는 기능에 월 구독료를 낼 가능성이 있다.

### 3순위: 교육 사업자와 코치

니즈:

- 글쓰기 챌린지
- 유튜브 성장 챌린지
- 틱톡 콘텐츠 업로드 챌린지
- 습관 인증 챌린지
- 수강생 과제 제출과 검수

과금 가능성:

- 중간

이유:

- 챌린지 완주율이 교육 상품의 성과와 연결된다.
- 다만 초기 지불 여력은 브랜드와 대행사보다 낮을 수 있다.

### 4순위: 일반 참가자

니즈:

- 챌린지 참여
- 진행률 확인
- 랭킹 확인
- 리워드 수령

과금 가능성:

- 낮음

이유:

- 참가자는 플랫폼 사용료보다 챌린지 참가비에 민감하다.
- 초기에는 참가자에게 직접 과금하기보다 운영자와 브랜드가 비용을 부담하는 구조가 낫다.

## 18.3 추천 수익 모델

## 18.3.1 1차 수익 모델: 챌린지 개설 수수료

운영자가 챌린지를 만들 때 건당 비용을 지불한다.

예시:

- 무료: 테스트 챌린지 1개, 참가자 10명까지
- 베이직: 챌린지 1개당 29,000원
- 프로: 챌린지 1개당 99,000원, 참가자 100명까지
- 브랜드: 챌린지 1개당 299,000원 이상, 리포트 포함

장점:

- 구현이 단순하다.
- 참가비 재분배와 직접 연결되지 않아 리스크가 낮다.
- 운영자가 챌린지 단위로 비용을 이해하기 쉽다.

단점:

- 반복 사용 전까지 매출 예측이 어렵다.
- 초기에는 무료 체험이 필요할 수 있다.

MVP 반영:

- 실제 결제 기능은 붙이지 않아도 된다.
- 요금제 안내와 관리자 수동 플랜 부여만 먼저 구현한다.

## 18.3.2 2차 수익 모델: 운영자 월 구독

반복 운영자에게 월 구독료를 받는다.

예시:

- Free: 챌린지 1개, 참가자 10명, CSV 제한
- Starter: 월 29,000원, 챌린지 3개, 참가자 100명
- Pro: 월 99,000원, 챌린지 20개, 참가자 1,000명, 리포트 포함
- Agency: 월 299,000원 이상, 브랜드/클라이언트 관리, 다중 운영자 권한

장점:

- 반복 매출을 만들 수 있다.
- SaaS 지표 관리가 쉽다.
- 대행사와 교육 사업자에게 적합하다.

단점:

- 무료 대비 명확한 기능 차이가 필요하다.
- 초기에 과금 장벽이 생길 수 있다.

MVP 반영:

- `plan_type`, `challenge_limit`, `participant_limit`, `csv_export_limit` 필드를 추가할 수 있다.
- 결제 연동 전에는 관리자가 수동으로 플랜을 변경한다.

## 18.3.3 3차 수익 모델: 브랜드 캠페인 운영 패키지

플랫폼 사용료와 운영 대행을 묶어 판매한다.

예시:

- 소형 캠페인: 50만 원부터
- 중형 캠페인: 200만 원부터
- 대형 캠페인: 별도 견적

포함 가능 항목:

- 챌린지 세팅
- 참가자 모집 페이지 구성
- 제출물 검수
- 성과 리포트
- 광고 표시 체크
- 정산 대상자 정리

장점:

- 초기 매출을 만들기 가장 쉽다.
- 제품이 완전하지 않아도 서비스형으로 보완할 수 있다.
- 고객 요구사항을 빠르게 학습할 수 있다.

단점:

- 인력 투입이 필요하다.
- 순수 SaaS보다 확장성이 낮다.

MVP 반영:

- 관리자와 운영자가 실제 운영을 수동으로 도와줄 수 있는 기능을 우선 만든다.
- 리포트 다운로드와 검수 로그가 중요하다.

## 18.3.4 4차 수익 모델: 거래 수수료

참가비, 리워드, 상금 지급액에 대해 일정 비율을 수수료로 받는 방식이다.

예시:

- 참가비 결제액의 3~10%
- 리워드 지급액의 3~5%
- 브랜드 캠페인 예산의 10~20%

장점:

- 챌린지가 성장할수록 매출도 같이 증가한다.
- 브랜드 캠페인에는 자연스러운 과금 방식이다.

단점:

- PG, 환불, 세무, 전자금융, 상금 지급 리스크가 커진다.
- 참가비 재분배 구조와 결합하면 법무 검토가 필수다.

MVP 반영:

- 적용하지 않는다.
- 2차 제품에서 PG와 법무 검토 후 도입한다.

## 18.3.5 5차 수익 모델: 리포트와 검증 기능 유료화

운영자에게 고급 리포트와 검증 기능을 유료로 제공한다.

유료 기능 예시:

- 챌린지 성과 리포트 PDF
- 제출물 증빙 압축 다운로드
- 검수 로그 리포트
- 참가자별 성과 분석
- 광고 표시 누락 체크리스트
- 유튜브 API 연동
- 틱톡 API 연동
- 브랜드 제출용 결과 보고서

장점:

- 브랜드와 대행사에 지불 명분이 크다.
- 기본 기능을 무료로 열어도 고급 기능에서 과금할 수 있다.

단점:

- 리포트 품질이 낮으면 유료 전환이 어렵다.

## 18.4 수익화 우선순위

초기에는 다음 순서가 가장 현실적이다.

1. 브랜드 캠페인 운영 패키지
2. 챌린지 개설 수수료
3. 운영자 월 구독
4. 고급 리포트 유료화
5. PG 결제 연동 후 거래 수수료

이 순서가 적합한 이유:

- 초기에는 제품 기능이 부족하므로 운영 대행과 결합해야 매출화가 빠르다.
- 개설 수수료와 구독료는 법무 리스크가 낮다.
- 거래 수수료는 매력적이지만, 결제와 상금 구조가 확정된 뒤 도입해야 한다.

## 18.5 MVP에 추가해야 할 수익화 기능

MVP에서 실제 결제는 붙이지 않더라도, 수익화를 검증하려면 다음 기능은 데이터 구조에 포함하는 것이 좋다.

필수:

- 운영자 플랜 구분: Free, Starter, Pro, Agency
- 챌린지 개설 가능 수 제한
- 참가자 수 제한
- CSV 다운로드 가능 여부
- 브랜드 리포트 사용 가능 여부
- 관리자 수동 플랜 변경

권장:

- 요금제 안내 페이지
- 플랜별 기능 비교
- 사용량 카운터
- 업그레이드 문의 버튼
- 캠페인 운영 문의 폼

MVP에서 제외:

- 카드 결제
- 자동 구독 결제
- 세금계산서 자동 발행
- PG 정산

## 18.6 수익화 검증 지표

제품 성공 지표와 별도로 수익화 지표를 추적해야 한다.

- 무료 운영자 가입 수
- 챌린지 개설 완료율
- 두 번째 챌린지 개설률
- 운영자당 월간 챌린지 수
- 챌린지당 평균 참가자 수
- CSV 다운로드 사용률
- 리포트 다운로드 사용률
- 업그레이드 문의 전환율
- 브랜드 캠페인 문의 수
- 유료 전환율
- 운영자 월 이탈률
- 운영자당 월 매출
- 캠페인당 평균 매출

## 18.7 초기 가격 가설

초기 가격은 너무 복잡하게 시작하지 않는다.

권장 가격 가설:

| 플랜 | 가격 | 대상 | 제한 |
| --- | --- | --- | --- |
| Free | 0원 | 테스트 운영자 | 챌린지 1개, 참가자 10명, CSV 제한 |
| Starter | 월 29,000원 | 소규모 운영자 | 챌린지 3개, 참가자 100명 |
| Pro | 월 99,000원 | 교육 사업자, 소형 브랜드 | 챌린지 20개, 참가자 1,000명, CSV/리포트 |
| Agency | 월 299,000원 이상 | 대행사, 브랜드 | 다중 운영자, 브랜드 리포트, 우선 지원 |
| Campaign | 건별 견적 | 브랜드 캠페인 | 운영 대행, 검수, 리포트 포함 |

가격은 확정값이 아니라 시장 검증용 가설이다. 첫 고객 5~10곳과 운영 테스트를 하며 조정한다.

## 18.8 수익화 리스크

| 리스크 | 설명 | 대응 |
| --- | --- | --- |
| 참가자가 돈을 내지 않을 가능성 | 참가자는 플랫폼보다 챌린지 자체에 비용을 지불하려 한다. | 참가자 과금보다 운영자 과금 중심으로 설계 |
| 운영자가 무료 도구로 버틸 가능성 | 초기 운영자는 스프레드시트와 폼으로 대체 가능하다. | 검수 시간 절감, 랭킹 자동화, 증빙 리포트로 차별화 |
| 브랜드 캠페인 리스크 | 광고 표시 누락, 허위 인증, 지급 지연 문제가 생길 수 있다. | 광고 표시 체크, 검수 로그, 지급 상태 관리 제공 |
| 거래 수수료 법무 리스크 | 참가비와 상금 재분배 구조는 법무 검토가 필요하다. | MVP에서는 거래 수수료 제외 |
| 대행형 매출의 확장성 한계 | 운영 대행은 인력이 필요하다. | 초기에 학습용으로 활용하고 반복 업무를 제품화 |

## 18.9 수익화 관점의 제품 포지셔닝

일반 참가자를 위한 챌린지 앱으로 포지셔닝하면 수익화가 약하다. 운영자와 브랜드를 위한 캠페인 운영 도구로 포지셔닝해야 돈을 받을 수 있다.

권장 포지셔닝:

"블로그, 유튜브, 틱톡 챌린지를 운영하는 브랜드와 운영자를 위한 인증 검수, 랭킹, 리포트, 정산 보조 SaaS"

피해야 할 포지셔닝:

- 참가비를 모아 상금을 나눠주는 앱
- 단순 챌린지 커뮤니티
- 리워드 앱
- 조회수 자동 수집 툴

## 18.10 PRD 반영 사항

수익화를 위해 PRD에 다음 기능을 추가한다.

데이터 모델 추가:

- `plans`
- `organizer_subscriptions`
- `usage_counters`
- `campaign_inquiries`

화면 추가:

- 요금제 안내
- 업그레이드 문의
- 운영자 사용량 표시
- 브랜드 캠페인 문의

관리자 기능 추가:

- 운영자 플랜 수동 변경
- 사용량 확인
- 캠페인 문의 확인
- 유료 전환 후보 운영자 표시

MVP 구현 방식:

- 실제 결제는 붙이지 않는다.
- 관리자가 플랜을 수동 부여한다.
- 요금제 페이지와 문의 폼으로 유료 의향을 먼저 검증한다.

## 19. 개발 전 반드시 결정할 사항

- MVP에서 참가비를 화면에 표시만 할지, 외부 입금 안내까지 할지
- 보상 방식을 어떤 문구로 표시할지
- 광고 표시 필요 챌린지의 기본값을 무엇으로 둘지
- 제출 마감 이후 보완 제출을 허용할지
- 랭킹 확정 후 검수 변경 프로세스
- 이미지 보관 기간
- 운영자 권한 부여 방식
- 개인정보 처리방침과 이용약관 초안
- Free, Starter, Pro, Agency 플랜 제한값
- 요금제 페이지를 공개할지 문의형으로 둘지
- 첫 유료 고객 대상: 브랜드, 대행사, 교육 사업자 중 어디부터 공략할지

## 20. 구현 우선순위 결론

가장 먼저 만들어야 하는 것은 결제나 상금 기능이 아니다. 1차 구현의 핵심은 다음 5개다.

1. 챌린지 생성
2. 참가 신청
3. 링크/사진 제출
4. 운영자 검수
5. 성공자와 랭킹 확정

이 5개가 안정적으로 동작하면 정산 보조, 이의 신청, 알림을 붙인다. PG 결제, 자동 송금, 외부 API 검수는 실제 운영 데이터와 법무·세무 검토 후 2차 제품으로 분리한다.

수익화 관점에서는 이 5개에 더해 다음 3개를 빠르게 붙인다.

1. 요금제 안내
2. 운영자 사용량 제한
3. 브랜드 캠페인 문의 폼

이렇게 해야 제품 개발과 동시에 유료 전환 가능성을 검증할 수 있다.
