# 챌린지 매니저 플랫폼 PRD

작성일: 2026-06-11  
문서 상태: MVP 개발 착수용 초안  
참고 문서: `challenge-manager-platform-plan.md`  
제품 범위: 블로그, 유튜브, 틱톡 챌린지의 생성, 참가, 인증 제출, 검수, 성공자 확정, 랭킹, 정산 보조

## 1. 제품 정의

챌린지 매니저 플랫폼은 운영자가 콘텐츠 기반 챌린지를 만들고, 참가자가 링크와 사진으로 미션을 인증하며, 운영자가 제출물을 검수해 성공자와 랭킹을 확정하는 챌린지 운영 플랫폼이다.

MVP에서는 참가비 자동 재분배, 상금 자동 송금, 세금 자동 계산, 외부 플랫폼 API 기반 자동 검수는 구현하지 않는다. 대신 운영자가 수동으로 결제 상태와 지급 상태를 관리할 수 있는 정산 보조 기능을 제공한다.

## 2. MVP 목표

## 2.1 핵심 목표

- 운영자가 챌린지를 만들 수 있다.
- 참가자가 챌린지에 참가 신청할 수 있다.
- 참가자가 미션별로 링크와 사진을 제출할 수 있다.
- 운영자가 제출물을 승인, 반려, 보완 요청할 수 있다.
- 시스템이 승인된 제출물을 기준으로 진행률과 성공 여부를 계산한다.
- 운영자가 성공자와 랭킹을 확정할 수 있다.
- 운영자가 정산 대상자 목록을 CSV로 다운로드할 수 있다.

## 2.2 MVP에서 검증할 가설

- 운영자는 기존 스프레드시트보다 편하게 참가자 인증을 관리할 수 있다.
- 참가자는 챌린지 조건과 제출 상태를 명확히 확인하면 완주율이 높아진다.
- 링크와 사진 기반 수동 검수만으로도 초기 챌린지 운영이 가능하다.
- 자동 송금 없이도 정산 대상자 확정과 CSV 다운로드만으로 초기 운영이 가능하다.

## 3. MVP 제외 범위

아래 기능은 MVP에서 제외한다.

- 참가비 풀 자동 재분배
- 상금 자동 송금
- 세금 자동 계산
- 유튜브 API 자동 조회수 수집
- 틱톡 API 자동 검수
- 블로그 본문 자동 크롤링
- AI 기반 자동 승인
- 카카오 알림톡
- 앱 푸시
- 포인트 지갑
- 참가자 간 커뮤니티 피드
- 고급 통계 대시보드
- 배지와 레벨 시스템
- 운영자 구독 결제

## 4. 사용자 역할

## 4.1 플랫폼 관리자

전체 플랫폼을 관리한다.

권한:

- 전체 회원 조회
- 전체 챌린지 조회
- 챌린지 숨김 또는 중단
- 신고 및 이의 신청 조회
- 관리자 로그 조회

MVP에서는 플랫폼 관리자 화면을 최소 기능으로만 제공한다.

## 4.2 챌린지 운영자

챌린지를 생성하고 운영한다.

권한:

- 챌린지 생성
- 본인 챌린지 수정
- 참가자 목록 조회
- 제출물 검수
- 참가자 결제 상태 수동 변경
- 성공자 확정
- 랭킹 확정
- 정산 CSV 다운로드

## 4.3 참가자

챌린지에 참가하고 미션을 제출한다.

권한:

- 챌린지 목록 조회
- 챌린지 상세 조회
- 참가 신청
- 내 참가 상태 조회
- 미션 제출
- 제출 이력 조회
- 반려 사유 확인
- 보완 제출
- 내 랭킹 조회

## 5. 핵심 사용자 시나리오

## 5.1 운영자 시나리오

1. 운영자가 로그인한다.
2. 챌린지 만들기 화면으로 이동한다.
3. 챌린지 제목, 설명, 기간, 모집 인원, 참가비 표시 금액, 성공 기준, 미션 수를 입력한다.
4. 제출 방식으로 링크, 사진, 설명 텍스트를 선택한다.
5. 보상 방식을 선택한다.
6. 챌린지를 저장하고 공개한다.
7. 참가자 신청 현황을 확인한다.
8. 결제 완료 여부를 수동으로 표시한다.
9. 제출물 검수 화면에서 승인, 반려, 보완 요청을 처리한다.
10. 챌린지 종료 후 성공자와 랭킹을 확인한다.
11. 랭킹을 확정한다.
12. 지급 대상자 CSV를 다운로드한다.

## 5.2 참가자 시나리오

1. 참가자가 챌린지 목록을 확인한다.
2. 챌린지 상세 페이지에서 기간, 성공 조건, 참가비, 보상 방식, 제출 방법을 확인한다.
3. 약관, 환불 정책, 개인정보 수집, 광고 표시 안내에 동의한다.
4. 참가 신청을 한다.
5. 운영자가 결제 완료 처리하면 참가 상태가 활성화된다.
6. 참가자는 미션별로 링크와 사진을 제출한다.
7. 제출물이 승인되면 진행률이 올라간다.
8. 반려 또는 보완 요청이 오면 사유를 확인하고 재제출한다.
9. 챌린지 종료 후 성공 여부와 랭킹을 확인한다.

## 6. 기능 요구사항

## 6.1 인증 및 계정

### 요구사항

- 이메일과 비밀번호로 회원가입할 수 있어야 한다.
- 이메일과 비밀번호로 로그인할 수 있어야 한다.
- 사용자는 참가자 역할을 기본으로 가진다.
- 운영자 권한은 플랫폼 관리자가 부여하거나, MVP에서는 관리자 시드 계정으로 제공한다.

### 우선순위

필수

### 수용 기준

- 사용자는 회원가입 후 로그인할 수 있다.
- 로그인하지 않은 사용자는 참가 신청, 미션 제출, 운영자 기능에 접근할 수 없다.
- 운영자 권한이 없는 사용자는 챌린지를 생성할 수 없다.

## 6.2 챌린지 생성

### 입력 항목

- 챌린지 제목
- 챌린지 설명
- 플랫폼 유형: 블로그, 유튜브, 틱톡, 기타
- 대표 이미지
- 모집 시작일
- 모집 종료일
- 챌린지 시작일
- 챌린지 종료일
- 최대 참가 인원
- 참가비 표시 금액
- 보상 방식
- 성공 기준
- 총 미션 수
- 최소 승인 미션 수
- 제출 마감 기준
- 광고 표시 필요 여부
- 환불 정책 설명
- 검수 기준 설명
- 공개 여부

### 보상 방식 MVP 옵션

- 무료 참가 + 운영자 고정 리워드
- 유료 참가 + 보증금 환급형
- 유료 참가 + 운영자 별도 리워드
- 기타 수동 정산

MVP에서는 보상 방식은 표시와 정산 참고용으로만 사용한다. 자동 송금이나 참가비 재분배 계산은 하지 않는다.

### 우선순위

필수

### 수용 기준

- 운영자는 필수 항목을 입력해 챌린지를 생성할 수 있다.
- 날짜가 논리적으로 맞지 않으면 저장할 수 없다.
- 최소 승인 미션 수는 총 미션 수보다 클 수 없다.
- 공개 챌린지는 참가자 목록 화면에 노출된다.
- 비공개 챌린지는 운영자만 볼 수 있다.

## 6.3 챌린지 목록 및 상세

### 목록 정보

- 챌린지 제목
- 플랫폼 유형
- 대표 이미지
- 모집 상태
- 진행 상태
- 모집 기간
- 진행 기간
- 참가비 표시 금액
- 현재 참가자 수
- 최대 참가자 수

### 상세 정보

- 챌린지 소개
- 성공 기준
- 제출 방법
- 검수 기준
- 광고 표시 필요 여부
- 환불 정책
- 보상 방식
- 참가 신청 버튼

### 우선순위

필수

### 수용 기준

- 참가자는 공개된 챌린지 목록을 볼 수 있다.
- 참가자는 챌린지 상세 정보를 볼 수 있다.
- 모집 기간이 아닌 챌린지는 참가 신청 버튼이 비활성화된다.
- 최대 참가 인원에 도달하면 참가 신청 버튼이 비활성화된다.

## 6.4 참가 신청

### 요구사항

- 참가자는 모집 중인 챌린지에 참가 신청할 수 있다.
- 참가 신청 전 필수 동의를 받아야 한다.
- 참가 신청 후 상태는 `PENDING_PAYMENT`가 된다.
- 운영자가 결제 완료를 수동으로 처리하면 `JOINED`가 된다.

### 필수 동의

- 서비스 이용 약관
- 챌린지 조건 확인
- 환불 정책 확인
- 개인정보 수집 및 이용
- 광고 표시 안내 확인

### 우선순위

필수

### 수용 기준

- 필수 동의를 하지 않으면 참가 신청할 수 없다.
- 같은 사용자는 같은 챌린지에 중복 신청할 수 없다.
- 최대 참가 인원을 초과해 신청할 수 없다.
- 참가 신청 이후 참가자는 내 챌린지 화면에서 상태를 확인할 수 있다.

## 6.5 결제 상태 수동 관리

### 요구사항

MVP에서는 PG 결제를 연동하지 않는다. 운영자가 외부 입금, 수동 결제, 테스트 결제 등 실제 운영 방식에 맞춰 결제 상태를 직접 변경한다.

### 결제 상태

- `PENDING_PAYMENT`: 결제 대기
- `PAID`: 결제 완료
- `CANCELED`: 결제 취소
- `REFUNDED`: 환불 완료

### 우선순위

필수

### 수용 기준

- 운영자는 본인 챌린지 참가자의 결제 상태를 변경할 수 있다.
- 결제 상태 변경 시 변경자, 변경 시각, 변경 전 상태, 변경 후 상태가 로그로 저장된다.
- 결제 완료 상태의 참가자만 미션을 제출할 수 있다.

## 6.6 미션 제출

### 제출 항목

- 제출 제목
- 콘텐츠 링크
- 인증 사진
- 설명 텍스트
- 플랫폼 계정명
- 해시태그

### 파일 정책

- 허용 확장자: JPG, PNG, WEBP
- 최대 파일 크기: 10MB
- 제출당 최대 이미지 수: 5개

### 우선순위

필수

### 수용 기준

- 참가자는 결제 완료 상태일 때만 미션을 제출할 수 있다.
- 참가자는 미션별 제출물을 작성할 수 있다.
- 링크 또는 사진 중 최소 하나는 필수다.
- 제출 완료 후 상태는 `SUBMITTED`가 된다.
- 참가자는 본인 제출물만 볼 수 있다.
- 운영자는 본인 챌린지의 모든 제출물을 볼 수 있다.

## 6.7 제출물 검수

### 검수 상태

- `SUBMITTED`: 제출 완료
- `APPROVED`: 승인
- `REJECTED`: 반려
- `NEEDS_REVISION`: 보완 요청

### 운영자 액션

- 승인
- 반려
- 보완 요청
- 검수 메모 입력

### 우선순위

필수

### 수용 기준

- 운영자는 본인 챌린지 제출물만 검수할 수 있다.
- 반려 또는 보완 요청 시 사유 입력이 필수다.
- 승인된 제출물만 진행률과 성공 여부에 반영된다.
- 검수 상태 변경 이력이 저장된다.
- 참가자는 반려 또는 보완 요청 사유를 확인할 수 있다.

## 6.8 보완 제출

### 요구사항

- `NEEDS_REVISION` 상태의 제출물은 참가자가 수정해 다시 제출할 수 있다.
- 재제출 시 기존 제출물 원본과 수정 이력을 보관한다.

### 우선순위

필수

### 수용 기준

- 참가자는 보완 요청 상태의 제출물을 수정할 수 있다.
- 재제출 후 상태는 `SUBMITTED`로 돌아간다.
- 운영자는 이전 제출 내용과 최신 제출 내용을 비교할 수 있다.

## 6.9 진행률 및 성공 여부 계산

### MVP 계산 방식

- 진행률 = 승인된 제출물 수 / 총 미션 수
- 성공 여부 = 승인된 제출물 수가 최소 승인 미션 수 이상이면 성공

### 우선순위

필수

### 수용 기준

- 참가자는 내 진행률을 확인할 수 있다.
- 운영자는 참가자별 승인 수와 성공 예상 여부를 확인할 수 있다.
- 성공 여부는 승인 상태 변경 시 자동으로 다시 계산된다.

## 6.10 랭킹

### MVP 랭킹 공식

1. 승인된 제출물 수가 많은 순
2. 마지막 승인 제출 시간이 빠른 순
3. 반려 횟수가 적은 순
4. 참가 신청 시간이 빠른 순

조회수, 좋아요, 댓글 수 기반 랭킹은 MVP에서 제외한다.

### 우선순위

필수

### 수용 기준

- 참가자는 챌린지 내 랭킹을 볼 수 있다.
- 운영자는 랭킹을 볼 수 있다.
- 운영자는 챌린지 종료 후 랭킹을 확정할 수 있다.
- 랭킹 확정 후에는 일반 검수 변경이 제한된다.

## 6.11 정산 보조

### 요구사항

MVP에서는 실제 지급을 자동화하지 않는다. 성공자와 지급 대상자를 확정하고 CSV로 내보내는 기능만 제공한다.

### 정산 목록 항목

- 챌린지명
- 참가자 이름 또는 닉네임
- 이메일
- 참가 상태
- 결제 상태
- 성공 여부
- 최종 순위
- 보상 방식
- 지급 예정 금액
- 지급 상태
- 지급 메모

### 지급 상태

- `PENDING`: 지급 대기
- `HOLD`: 지급 보류
- `PAID`: 지급 완료

- `CANCELED`: 지급 취소

### 우선순위

필수

## 6.12 이의 신청

### 요구사항

- 이의 신청 상태 변경 이력이 저장된다.

## 6.13 알림

### MVP 알림 방식

앱 푸시, 문자, 카카오 알림톡은 제외한다. MVP에서는 웹 화면 내부 알림과 이메일 알림만 고려한다. 이메일 연동이 부담되면 웹 알림만 먼저 구현한다.

### 알림 이벤트

- 참가 신청 완료
- 결제 완료 처리
- 제출물 승인
- 제출물 반려
- 보완 요청
- 랭킹 확정
- 지급 상태 변경

### 우선순위

권장

### 수용 기준

- 사용자는 본인 관련 알림만 볼 수 있다.
- 읽음 여부가 저장된다.

## 10. 비기능 요구사항

## 10.1 보안

- 비밀번호는 해시로 저장한다.
- 운영자와 참가자 권한을 분리한다.
- 사용자는 본인 데이터만 조회할 수 있어야 한다.
- 운영자는 본인 챌린지 데이터만 관리할 수 있어야 한다.
- 관리자 작업은 로그로 남긴다.

## 10.2 개인정보

- MVP에서는 필요한 개인정보만 수집한다.
- 지급 자동화 전에는 주민등록번호 등 고위험 개인정보를 수집하지 않는다.
- 환불 계좌 정보는 MVP에서 수집하지 않는 것을 권장한다.
- 개인정보 처리방침과 약관 버전을 동의 기록에 저장한다.

## 10.3 파일 업로드

- 이미지 확장자와 MIME 타입을 검증한다.
- 파일 크기를 제한한다.
- 업로드 파일은 공개 URL 접근 권한을 신중히 설정한다.
- 삭제된 제출물의 파일 보관 정책을 정해야 한다.

## 10.4 감사 가능성

- 검수, 결제 상태, 지급 상태 변경은 모두 추적 가능해야 한다.
- 랭킹 확정 후 변경은 제한하거나 별도 로그를 남긴다.

## 11. MVP 성공 지표

1차 운영 기준으로 다음 지표를 본다.

- 생성된 챌린지 수
- 챌린지당 참가 신청 수
- 참가 신청 후 결제 완료 처리율
- 참가자당 평균 제출 수
- 제출 승인률
- 반려 후 재제출률
- 챌린지 성공률
- 운영자 평균 검수 시간
- 정산 CSV 다운로드 횟수
- 운영자 재사용률

## 13. 주요 정책 결정

개발 시작 전에 다음 결정을 확정해야 한다.

- MVP에서 실제 PG 결제를 붙일지 여부
- 참가비를 서비스 이용료, 보증금, 리워드 재원 중 무엇으로 표현할지
- 보상 방식별 약관 문구
- 광고 표시가 필요한 챌린지의 기준
- 미성년자 참여 허용 여부
- 이미지 보관 기간
- 제출 마감 이후 보완 제출 허용 여부
- 랭킹 확정 후 검수 변경 가능 여부
- 운영자 권한을 누가 부여할지

## 14. 오픈 이슈

- 법무 검토 전까지 참가비 기반 상금 재분배는 자동화하지 않는다.
- 세무 검토 전까지 현금 지급 자동화는 하지 않는다.
- 외부 플랫폼 API 연동은 MVP 이후로 미룬다.
- 초기에는 수동 운영을 감수하고 검수와 정산 흐름의 사용성을 먼저 검증한다.

## 15. 최종 MVP 정의

MVP의 최종 정의는 다음과 같다.

"운영자가 블로그, 유튜브, 틱톡 챌린지를 만들고, 참가자가 링크와 사진으로 미션을 제출하며, 운영자가 이를 검수해 성공자와 랭킹을 확정하고 정산 대상자 목록을 다운로드할 수 있는 웹 플랫폼"

이 범위는 개발 가능성이 높고, 법무·세무·PG 리스크를 통제하면서도 실제 챌린지 운영 가치를 검증할 수 있다.
